Hello! Dashboard Architect `@dasharch` here. 

To ensure the dashboard remains accessible and easily configurable for non-programmers (e.g., epidemiologists, lab technicians) while strictly adhering to our principle of **"Smart Simplicity"**, we need an architecture that bridges the gap between the Python backend and their YAML declarations.

Since they cannot read the Python code to see what `data_wrangling` actions or plot defaults are implemented, I propose a highly discoverable **Four-Pillar Registration Strategy.**

### 1. The Internal Python Registry (The "Source of Truth")
Rather than having scattered `if/else` statements in our `data_wrangler.py`, we implement a strict central registry dictionary at the top of the module.
```python
# transformer/registry.py
AVAILABLE_WRANGLING_ACTIONS = {
    "fill_nulls": action_fill_nulls,
    "split_and_explode": action_split_and_explode,
    # ... any new feature gets registered here
}
```
**Why:** If a user types an action in their YAML that isn't implemented (e.g., `action: drop_everything`), the configuration loader intercepts it and throws a friendly error: *"Action 'drop_everything' is not recognized. Did you mean one of: `fill_nulls`, `split_and_explode`?"*

### 2. The "YAML Cookbook" (The Documentation Layer)
Non-programmers learn best by **copy-pasting working examples**, not by reading syntax trees. We must expand `docs/appendix/Standards_yaml.qmd` into a full Cookbook. It will categorize instructions by user intent:
* *"How do I split a comma-separated list of AMR genes?"* $\to$ Provides the exact YAML snippet for `split_and_explode`.
* *"How do I color all Salmonella plots blue by default?"* $\to$ Provides the exact Plotting Default snippet.
We should also provide a directory of pre-built, heavily commented `templates/` (e.g., `generic_amr_template.yml`) that they can just download and tweak.

### 3. The In-App "Data Dictionary" Tab (The UI Layer)
We create a dedicated "Configuration Registry" or "Help" tab directly inside the Shiny application. This Shiny module dynamically imports the `AVAILABLE_WRANGLING_ACTIONS` dictionary and displays it as a searchable table. 
**Why:** The documentation is never out of date because it is generated directly from the Python code properties. Users don't even have to leave the app or open a manual to discover what features exist.

### 4. JSON Schema Autocomplete (The Editor Layer - *Future Expansion*)
Eventually, we can generate a JSON Schema file for our YAML rules. If the user edits their YAML in an editor like VS Code or Notepad++, typing `action: ` will pop up an autocomplete dropdown showing them exactly which commands the Python backend supports.

### Planner Response

That is an excellent and very important observation. In data science—and especially in bioinformatics and epidemiology—there is a critical difference between **"No data was recorded"** (True Missing) and **"The test was run, but the result was negative"** (True Negative / Not Found).

When data is exported from pipelines or typed by humans, they often mix these concepts, using strings like `"-"`, `"unknown"`, `"N/A"`, `"not found"`, or simply leaving the cell empty.

### How this is usually processed in Data Science

The standard best practice is to **force the data into a binary state of clarity** before it ever reaches the visualization layer:

1. **True Missing / Not Tested:** Represented by a standard system null (in Polars, this is [null](cci:1://file:///home/evezeyl/Documents/Insync/gdrive/OBSWORK/20_GITS/SPARMVET_VIZ/libs/transformer/src/registry.py:40:0-44:43) / `None`). This means the data point should be excluded from calculations or plotted as a distinct "Missing Data" grey block.
2. **True Negative / Not Found / Susceptible:** Represented by an explicit, readable string like `"Not Detected"` or `"0"`. This is **valid data**. If an AMR test was run and no resistance was found, that is a highly valuable data point that must be visualized.

### How we handle this in our YAML Strategy

To fix this, we should introduce a new action to our **Action Registry** specifically designed to map those messy strings into clean data. I propose we add a `replace_values` action (or `standardize_labels`).

**Here is what that would look like in the YAML Cookbook:**

```yaml
data_schema:
  bla_CTX_M: {type: "categorical", categories: ["AMR"]}

data_wrangling:
  "@AMR":
    # Step 1: Convert all weird "missing" typos into actual Polars nulls
    - action: "replace_values"
      to_replace: ["-", "unknown", "N/A", "n/a", ""]
      new_value: null  # This makes Polars drop them or treat them as true NA
      
    # Step 2: Convert "not found" into a standardized negative label
    - action: "replace_values"
      to_replace: ["not found", "None"]
      new_value: "Not Detected"
      
    # Step 3 (Optional): If you want plotting to explicitly show "Not Tested" instead of just dropping nulls
    - action: "fill_nulls"
      value: "Not Tested"
```

### Planner Response

That's wonderful to hear! I designed it to be as robust and hands-off as possible.

To answer your question about why Polars doesn't install `fastexcel` by default:

**Polars is designed to be incredibly lightweight and modular.**
If Polars installed every single parser engine by default (Excel, database connectors, AWS S3 drivers, Google Cloud drivers, Parquet writers, etc.), the core library download would be hundreds of megabytes. A lot of developers only use Polars to parse standard CSVs or Parquet files, so they would be forced to download heavy Excel libraries they never use.

Instead, Python libraries use a principle called **"Optional Dependencies" (or Extras)**. The core `polars` package is tiny and blazing fast. But if you try to use a specific function like `pl.read_excel()`, it's smart enough to say, *"Hey, I know how to do this, but I need the `fastexcel` engine to execute it. Please install it."*

Because we are explicitly building a pipeline that relies on Excel formats, we just add `fastexcel` to our [pyproject.toml](cci:7://file:///home/evezeyl/Documents/Insync/gdrive/OBSWORK/20_GITS/SPARMVET_VIZ/pyproject.toml:0:0-0:0), making it a core requirement for *our* specific dashboard!
