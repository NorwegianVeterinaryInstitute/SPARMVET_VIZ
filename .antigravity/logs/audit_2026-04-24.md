## Session Block 1 — 2026-04-24 — @dasharch — Multi-System Deployment Architecture (ADR-048)

### Context
Architectural discussion on how SPARMVET_VIZ will be deployed across multiple platforms: Galaxy (GxIT), IRIDA (REST API), institutional servers, and local developer machines — all using a single Docker image.

### Key Decisions

**1. Deployment Profile replaces connector YAML**
- Current `config/connectors/local/local_connector.yaml` is extended to a full **Deployment Profile** schema.
- New fields: `deployment_type` (filesystem / irida / galaxy), `default_manifest`, `default_persona`, `project_root`, `deployment_name`, optional `irida` block.
- Directory rename `config/connectors/` → `config/deployment/` deferred to Phase 23 (requires Bootloader code change).

**2. Bootloader resolution chain (4 levels, first match wins)**
1. `SPARMVET_PROFILE` env var — used by Galaxy XML wrapper, IRIDA container launch, Docker Compose, systemd
2. `~/.sparmvet/profile.yaml` — user-level (local PC, no env var needed)
3. `/etc/sparmvet/profile.yaml` — system-level (server, admin-placed)
4. `config/connectors/local/local_connector.yaml` — dev fallback (current)

**3. Deployment contexts confirmed**
- **Galaxy**: XML wrapper per pipeline sets `SPARMVET_PROFILE`; mounts Galaxy job dir as `project_root`; same image, different wrappers.
- **IRIDA**: REST API only (no env var injection like Galaxy). `IridaConnector` fetches data via OAuth2 (`SPARMVET_IRIDA_TOKEN` env var, never in profile) to local cache; app then sees filesystem paths.
- **General server**: admin places `/etc/sparmvet/profile.yaml`; Docker Compose with `SPARMVET_PROFILE` for multiple pipeline deployments.
- **Local PC**: dev fallback or `~/.sparmvet/profile.yaml`.

**4. Connector abstraction library (Phase 23)**
New `libs/connectors/` with `BaseConnector` interface (`resolve_paths`, `fetch_data`, `get_manifest_path`, `get_default_persona`). Three implementations: `FilesystemConnector`, `IridaConnector`, `GalaxyConnector`. All return local filesystem paths after `fetch_data()` — rest of app is source-system-agnostic.

**5. Manifest vs Persona — orthogonal**
- `default_manifest` in profile = which pipeline (AMR, Plasmid, Virulence). Controlled by deployment.
- `default_persona` in profile = who the user is (access level). Controlled by deployment.
- NOT bound to persona template. Persona templates remain unchanged.

**6. Multiple pipelines on one image**
- Recommended: one deployment per pipeline (separate `SPARMVET_PROFILE` each).
- Alternative: one deployment, `default_manifest` absent, manifest selector shown (persona-gated).
