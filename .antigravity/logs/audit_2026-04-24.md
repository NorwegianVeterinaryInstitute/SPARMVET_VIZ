## Session Block 1 — 2026-04-24 — @dasharch — Multi-System Deployment Architecture (ADR-048)

### Context
Architectural discussion on how SPARMVET_VIZ will be deployed across multiple platforms: Galaxy (GxIT), IRIDA (REST API), institutional servers, and local developer machines — all using a single Docker image.

### Key Decisions

**1. Deployment Profile replaces connector YAML**
- Current `config/connectors/local/local_connector.yaml` is extended to a full **Deployment Profile** schema.
- New fields: `deployment_type` (filesystem / irida / galaxy), `default_manifest`, `default_persona`, `project_root`, `deployment_name`, optional `irida` block.
- Directory rename `config/connectors/` → `config/deployment/` deferred to Phase 23 (requires Bootloader code change).

**2. Bootloader resolution chain (4 levels, first match wins)**
1. `SPARMVET_PROFILE` env var — used by Galaxy XML wrapper, IRIDA container launch, Docker Compose, systemd
2. `~/.sparmvet/profile.yaml` — user-level (local PC, no env var needed)
3. `/etc/sparmvet/profile.yaml` — system-level (server, admin-placed)
4. `config/connectors/local/local_connector.yaml` — dev fallback (current)

**3. Deployment contexts confirmed**
- **Galaxy**: XML wrapper per pipeline sets `SPARMVET_PROFILE`; mounts Galaxy job dir as `project_root`; same image, different wrappers.
- **IRIDA**: REST API only (no env var injection like Galaxy). `IridaConnector` fetches data via OAuth2 (`SPARMVET_IRIDA_TOKEN` env var, never in profile) to local cache; app then sees filesystem paths.
- **General server**: admin places `/etc/sparmvet/profile.yaml`; Docker Compose with `SPARMVET_PROFILE` for multiple pipeline deployments.
- **Local PC**: dev fallback or `~/.sparmvet/profile.yaml`.

**4. Connector abstraction library (Phase 23)**
New `libs/connectors/` with `BaseConnector` interface (`resolve_paths`, `fetch_data`, `get_manifest_path`, `get_default_persona`). Three implementations: `FilesystemConnector`, `IridaConnector`, `GalaxyConnector`. All return local filesystem paths after `fetch_data()` — rest of app is source-system-agnostic.

**5. Manifest vs Persona — orthogonal**
- `default_manifest` in profile = which pipeline (AMR, Plasmid, Virulence). Controlled by deployment.
- `default_persona` in profile = who the user is (access level). Controlled by deployment.
- NOT bound to persona template. Persona templates remain unchanged.

**6. Multiple pipelines on one image**
- Recommended: one deployment per pipeline (separate `SPARMVET_PROFILE` each).
- Alternative: one deployment, `default_manifest` absent, manifest selector shown (persona-gated).

## Session Block 2 — 2026-04-24 — @dasharch — Conditional UI Logic Design (Left Sidebar)

### Context
Architectural design session resolving open questions about left sidebar conditional UI logic: manifest selection, metadata ingestion, export sub-features, session management, data ingestion, and panel-context dependency of the sidebar.

### Key Decisions

**1. Metadata Ingestion (§9 of ui_implementation_contract.md)**
- Full replacement model — uploaded file becomes new source of truth for that metadata source.
- Triggers T1 rebuild via DataOrchestrator after MetadataValidator gate passes.
- Provenance: original uploaded filename stored in Location 1 sidecar (`metadata_provenance.yaml`) and included in all subsequent export bundles (README.txt, report.qmd).
- Persona gate: ≥ `pipeline_exploration_advanced` (`metadata_ingestion_enabled: true`).
- Status: deferred Phase 22.

**2. Export label rename**
- `export_user_name` field label changed to "Bundle label / name" — clarifies it is not a username.
- Minor code fix pending (home_theater.py).

**3. Export Active Graph (§7.3)**
- Single-plot export: plot file + recipe fragment + FILTERS.txt.
- Persona gate: ≥ `pipeline_exploration_advanced` (`export_graph_enabled: true`).
- Status: deferred Phase 22.

**4. Export location**
- Browser download (= local PC / client) for all deployment contexts — works identically in Galaxy, IRIDA, and local.
- Server-side write to Location 4 deferred as enhancement for pipeline archiving.

**5. Session Save / Import (§7.1)**
- Named sessions (`.json` in Location 4 subdirectory). Multiple sessions per user (different pipeline runs / dates).
- Ghost save: `_autosave.json` single slot, triggered on every `btn_apply` and filter commit.

- Status: deferred Phase 22.

**6. Data Ingestion + Excel Converter (§10)**

- Multi-file upload with schema association dropdown (from manifests directory).
- Excel converter: sheet → role → schema assignment; writes TSVs to Location 1 via ExcelHandler (already exists).
- Status: deferred Phase 22.

**7. Left Sidebar Panel-Context Dependency (§11)**
- Formalized: sidebar content is physically replaced (not CSS-hidden) on panel switch.
- Home: Filter Recipe Builder + System Tools.
- Blueprint Architect: manifest/component navigation.
- Gallery: Focus Mode only.
- Dev Studio: TBD — deferred until Dev Studio design session.
- Blueprint Architect filter controls: undecided, deferred until Architect mode finalized.

**8. New persona template flags added to all 5 templates**
- `export_graph_enabled`, `metadata_ingestion_enabled`, `data_ingestion_enabled`
- Values per persona per matrix in rules_ui_dashboard.md §3.

## Session Block 3 — 2026-04-24 — @dasharch — Persona & Deployment Documentation Pass

### Context
Full documentation pass covering: UI persona system (full rewrite of ui_persona.qmd), deployment configuration (full rewrite of connector.qmd with naming decision), and a new internal developer rule for feature flag dependencies.

### Key Decisions

**Naming decision: `connectors/` → `deployment/`**
- Confirmed: the directory WILL be renamed in Phase 23 (one line in bootloader.py).
- Until Phase 23: `config/connectors/` remains active. All documentation uses future name `config/deployment/` to avoid re-documenting after rename.
- Template already uses new schema. Dev fallback (`local_connector.yaml`) remains minimal old schema until migration.

**Feature flag dependency groups formalized**
Four groups documented with explicit parent-gate relationships:
- Group A: Viewing — no dependencies (`export_bundle_enabled`)
- Group B: T3 Sandbox — `interactivity_enabled` is master gate for comparison, session, export_graph
- Group C: Data Ingestion — `import_helper_enabled` is gate for `data_ingestion_enabled`; `metadata_ingestion_enabled` semi-independent
- Group D: Developer — `developer_mode_enabled` gates Dev Studio; `gallery_enabled` independent

**PersonaManager resolution rules documented**
- If `interactivity_enabled: false` → force comparison/session/export_graph to false, log warning
- If `import_helper_enabled: false` → force `data_ingestion_enabled` to false, log warning
- Deployment profile `data_ingestion_enabled: false` overrides persona (but NOT `metadata_ingestion_enabled`)
- Right sidebar suppression: structural (layout exclusion), not flag-based
