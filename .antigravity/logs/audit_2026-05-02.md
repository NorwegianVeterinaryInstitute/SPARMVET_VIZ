### Batch 1 ‚Äî Pipeline persona bug fixes (commit `3b0d093`)

Three bugs prevented pipeline personas from loading any data:

**Bug 1: `bootloader.get_default_project()` wrong project.**
Previously returned the first filesystem glob result. Fixed to check `fixed_manifest` from the persona template first.
File: `app/src/bootloader.py` ‚Äî `get_default_project()`

**Bug 2: `server.active_cfg()` throws when manifest selector hidden.**
Was calling `input.project_id()` directly, which Shiny raises on when the input does not exist (hidden selector for pipeline-static / pipeline-exploration-simple). Fixed to use `_safe_input(input, "project_id", bootloader.get_default_project())`.
File: `app/src/server.py` ‚Äî `active_cfg()` reactive calc

**Bug 3: Filters accordion shown for pipeline-static.**
`home_theater.py` was always rendering the Filters accordion. Fixed: gates the accordion on `bootloader.is_enabled("interactivity_enabled")`. When false, the accordion is absent entirely (not hidden inside ‚Äî the panel itself is not created).
File: `app/handlers/home_theater.py`

### VizFactory integer axis breaks (VIZ-BREAKS-INT)

`libs/viz_factory/src/viz_factory/scales/core.py`:
- Added `_integer_breaks(lims)` helper: `MaxNLocator(integer=True).tick_values(lims[0], lims[1])`.
- Added `_resolve_continuous_spec(spec)`: pops `breaks_integer` key and substitutes `breaks=_integer_breaks`.
- Both `scale_x_continuous` and `scale_y_continuous` handlers call `_resolve_continuous_spec`.
- Note: `MaxNLocator` cannot be used directly as a plotnine breaks callable (plotnine calls `breaks(limits)` with a tuple). Must be wrapped.

Manifest side: `breaks_integer: true` must be nested under `params:` in the layer entry, not at the layer's top level (VizFactory reads `layer_spec.get('params', {})`).

### MetadataValidator dtype_map audit (prerequisite for IMPORT-1)

`libs/transformer/src/transformer/metadata_validator.py`:
- Removed `.lower()` from type-string lookup (was masking gaps by normalizing but not helping if key absent).
- Expanded `dtype_map` with all YAML type aliases: `numeric‚ÜíFloat64`, `character‚ÜíUtf8`, `date‚ÜíDate`, `integer‚ÜíInt64`, `bool/boolean‚ÜíBoolean`, PascalCase aliases (`Int64`, `Float64`, `String`, `Boolean`, `Date`, `Categorical`).
- Added `warnings.warn` for unrecognised type strings (was silently skipping).

Root cause: prior numeric filters cast columns to Utf8 as a workaround (e.g., `Float64` `2022.0` ‚Üí `"2022.0"` compared to filter value `"2022"` ‚Üí silent mismatch). Removing the cast exposed the dtype_map gap.
