# Design: Single Graph Export — Lineage Recipe + T3 Threading
# Covers: EXPORT-SGE-2 + future T3 node type gap
# Status: design only — NOT implemented. Read and discuss before building.

---

## Problem 1 — T3 node types not threaded through `_resolve_active_lf`

**Current state:**
Only `drop_column` and `filter_row`/`exclusion_row` T3 nodes are applied to the
LazyFrame before export. They are extracted by `_t3_drop_columns()` and
`_t3_filter_rows()` in `home_theater.py`.

If new T3 node types are added (rename, derive, pivot, cast …), they will:
- be committed to `t3_recipe.json` correctly (ghost + session)
- appear in the on-screen plot correctly IF `viz_factory.render()` respects them
- **NOT appear in the exported TSV** (`{safe_pid}_data.tsv`) in the single graph export

**Where the gap lives:**
`single_graph_export_handlers.py` — lines ~139–147:
```python
lf = _resolve_active_lf(spec)
drops = [c for c in _t3_drop_columns(subtab) if c in lf.collect_schema().names()]
if drops:
    lf = lf.drop(drops)
```
Filter rows are injected into `plot_spec["filters"]` (line ~119–131), which
`viz_factory.render()` applies internally. Column drops are applied here directly.

**Required fix when new T3 node types land:**
Extend `_resolve_active_lf` — or add a helper `_apply_t3_to_lf(lf, subtab)` in
`home_theater.py` — that handles ALL committed T3 node types for the given subtab.
Currently that helper does not exist; the logic is split between the export handler
and the render path. The render path applies T3 via `viz_factory` which may handle
types the export handler doesn't know about.

**Recommended approach:**
Create `_apply_t3_to_lf(lf, subtab) -> LazyFrame` in `home_theater.py` that iterates
`_active_plot_t3_nodes(subtab)` and dispatches on `node_type`:
- `drop_column` → `lf.drop([col])`
- `filter_row` / `exclusion_row` → already handled via `_apply_filter_rows`
- `rename` (future) → `lf.rename({old: new})`
- `derive` (future) → `lf.with_columns([expr])`
- `pivot` (future) → `lf.collect().pivot(...).lazy()` (pivot breaks LazyFrame)

Then use `_apply_t3_to_lf` in both the export handler and the data-preview path
so both are consistent.

## Problem 2 — EXPORT-SGE-2: no lineage recipe in the export bundle

**Current state:**
`manifest_fragment.yaml` contains only:
```yaml
plots:
  amr_profile:   # just the plot spec
    type: …
    filters: …
```

A reader cannot reconstruct the data pipeline from this alone. They need to know
what assembly/wrangling produced the dataset the plot reads.

**Data linkage pattern (from manifests):**
Plot spec has `target_dataset: "legacy_results"`.
The full manifest has:
- `data_schemas.legacy_results` — field contracts
- `assembly_manifests.legacy_results` — the T1→T2 assembly recipe (join, wrangle steps)

**Proposed bundle file: `full_recipe.yaml`**

```yaml
# Full reproducible recipe for plot: amr_profile
# Generated by SPARMVET-VIZ single graph export

plot_id: amr_profile
target_dataset: legacy_results

# ── T1/T2 ──────────────────────────────────────────────────────────────────
data_schema:
  # from manifest data_schemas.{target_dataset}
  fields: …

assembly:
  # from manifest assembly_manifests.{target_dataset}
  recipe: …

# ── T3 (user session) ───────────────────────────────────────────────────────
t3_nodes:
  # from committed home_state.t3_recipe_by_plot[subtab]
  - node_type: drop_column
    params: {column: year}
    active: true
  …

# ── Plot spec ───────────────────────────────────────────────────────────────
plot_spec:
  # from manifest plots.{plot_id}
  type: bar
  …
```

**Implementation sketch (in `single_graph_export_handlers.py`):**

```python
# After resolving spec and cfg:
target_ds = spec.get("target_dataset", "")
raw = cfg.raw_config

full_recipe = {
    "plot_id": p_id,
    "target_dataset": target_ds,
    "data_schema": raw.get("data_schemas", {}).get(target_ds, {}),
    "assembly": raw.get("assembly_manifests", {}).get(target_ds, {}),
    "t3_nodes": _active_plot_t3_nodes(subtab),
    "plot_spec": spec,
}
zf.writestr("full_recipe.yaml", yaml.safe_dump(full_recipe, sort_keys=False))
```

Keep `manifest_fragment.yaml` for backwards compatibility (it's simpler / smaller).
Add `full_recipe.yaml` as the new authoritative file.
Update `README.txt` to mention both files and their purpose.

**Edge cases to resolve before implementing:**
- What if `target_dataset` is absent from the plot spec? (fall back to empty assembly block)
- What if `assembly_manifests` uses `!include` tags? (raw_config may not have resolved them)
  → Check if `active_cfg().raw_config` returns resolved or raw YAML. If raw, the `!include`
    paths need to be resolved or the included file content inlined.
- Multi-dataset plots (multiple `target_dataset` values)? Check if the schema supports this.

## Summary — two independent tasks

| Task | File(s) | Complexity | Prerequisite |
|------|---------|-----------|-------------|
| T3 threading gap | `home_theater.py` (new `_apply_t3_to_lf`), `single_graph_export_handlers.py` | Low (now) / Medium (when new node types land) | Add new T3 node type first |
| SGE-2 lineage recipe | `single_graph_export_handlers.py` | Low-Medium | Confirm `!include` resolution in `active_cfg().raw_config` |

**Suggested order:** resolve `!include` question first (one grep / test), then implement
SGE-2 in a single PR. T3 threading gap is a forward-compatibility concern — no urgency
until the first rename/derive node type is designed.
