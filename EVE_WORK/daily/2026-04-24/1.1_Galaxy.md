## Galaxy Integration — Configuration Resolution Architecture

Here is the complete picture and the recommended solution:

### The Core Problem

Galaxy deploys **one app container image**, but needs to launch it for **multiple different pipelines**, each with a different default manifest. The mechanism Galaxy provides is:

1. **Environment variables** — injected at container start via the XML `<environment_variables>` block
2. **`<configfiles>`** — Galaxy generates a config file from a Cheetah template before launch, mounted into the container

### Recommended Solution: `SPARMVET_CONNECTOR` + `SPARMVET_MANIFEST` env vars

The Bootloader already has authority over path resolution (ADR-031). The cleanest extension is:

**At launch time, Galaxy injects two env vars:**

```xml
<environment_variables>
  <environment_variable name="SPARMVET_CONNECTOR">
      /path/to/galaxy_connector.yaml
  </environment_variable>
  <environment_variable name="SPARMVET_MANIFEST">
      /path/to/pipeline_X_manifest.yaml
  </environment_variable>
  <!-- optionally persona -->
  <environment_variable name="SPARMVET_PERSONA">pipeline-static</environment_variable>
</environment_variables>
```

**Bootloader resolution priority** (new rule to add):

```
1. SPARMVET_CONNECTOR env var          → overrides all
2. SPARMVET_MANIFEST env var           → overrides connector default_manifest
3. persona template default_manifest   → fallback
4. local_connector.yaml                → local dev fallback
```

This means:

- **Local dev**: no env vars, uses `local_connector.yaml` as today
- **Galaxy pipeline A** (e.g., AMR pipeline): env var points to `amr_connector.yaml` + `amr_manifest.yaml`
- **Galaxy pipeline B** (e.g., Plasmid pipeline): same image, different env vars → different manifest loaded

The connector YAML for Galaxy needs two additions:

```yaml
# galaxy_amr_connector.yaml
default_manifest: "/data/pipeline/amr/manifest.yaml"
default_persona: "pipeline-static"

locations:
  raw_data: "/galaxy/job_working_directory/inputs/"
  curated_data: "/galaxy/job_working_directory/parquet/"
  user_sessions: "/galaxy/user_data/sessions/"
  ...
```

### How Galaxy passes the Galaxy history dataset (the pipeline output)

A Galaxy pipeline tool deposits a Parquet or TSV result into the history. The interactive tool XML declares an `input` param of `type="data"` pointing at that history dataset. Galaxy mounts it into the container job directory. The connector's `raw_data` or `curated_data` location then points to that mounted path — so the app picks it up automatically without knowing it came from Galaxy.

```xml
<param name="input_results" type="data" format="parquet,tabular" label="Pipeline results"/>
```

The `SPARMVET_CONNECTOR` env var's `curated_data` path = where Galaxy mounted `${input_results}`.

### Why NOT per-persona default_manifest

Persona = **who the user is** (their access level). Manifest = **which pipeline/dataset** they're looking at. These are orthogonal. A `pipeline_static` user looking at AMR results should get the AMR manifest; the same persona looking at Plasmid results gets the Plasmid manifest. Binding manifest to persona would couple two independent dimensions.

### What this means for current architecture

|Layer|Change needed|
|---|---|
|`bootloader.py`|Add env var resolution: `SPARMVET_CONNECTOR`, `SPARMVET_MANIFEST`, `SPARMVET_PERSONA` with priority chain|
|`local_connector.yaml`|Add optional `default_manifest` key|
|Persona templates|**No change** — persona stays orthogonal to manifest|
|`ui_implementation_contract.md`|New §: Deployment Configuration Resolution|
|Galaxy wrapper (future)|XML tool file — one per pipeline, same container image|

### For the Persona → Manifest question specifically

The answer is: **connector-driven, not persona-driven**. Each Galaxy tool wrapper (one per pipeline) sets `SPARMVET_CONNECTOR` to a connector YAML that already knows its default manifest. Lower personas simply cannot change the manifest (the UI control is hidden by persona), but the manifest itself is selected by the connector/deployment context.
