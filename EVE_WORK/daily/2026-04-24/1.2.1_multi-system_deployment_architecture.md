## Multi-System Deployment Architecture

### What we've learned about each system

**Galaxy**: Env vars injected by XML wrapper at container launch. Data mounted into job directory. Standard GxIT mechanism.

**IRIDA**: Completely different model — **REST API only**, OAuth2 authentication. No env vars, no mounted volumes. IRIDA uses Galaxy for workflows internally, but external tools connect via API calls, not GxIT. This means IRIDA integration requires a **data fetcher** component that pulls files from IRIDA's REST API before (or instead of) reading from a local path.

**General server / local PC**: Pure filesystem. Admin configures paths. No external API.

---

### The fundamental split this creates

You have **two data access paradigms** that cannot be unified at the filesystem level:

|Paradigm|Systems|How data arrives|
|---|---|---|
|**Filesystem**|Local PC, general server, Galaxy (data mounted)|Paths in connector YAML|
|**API-fetched**|IRIDA, potentially future systems|OAuth2 + REST → download to local cache|

This means the connector cannot just be a path map — it also needs to describe **how to acquire data**, not just where it lives once acquired.

---

### The "pipeline module" idea — this is the key

Your instinct about a **small module per pipeline** that transmits the default manifest to the app is the right abstraction. Here is how I'd name and scope it:

**A "deployment profile"** — one YAML file per deployment instance, which declares:

1. **What system type** this is (`galaxy`, `irida`, `filesystem`)
2. **Data acquisition config** (paths for filesystem; API base URL + auth for IRIDA)
3. **Default manifest** (path or identifier)
4. **Default persona** (locks the UI for this deployment)
5. **Project root** (all relative locations resolve under this)

The Bootloader reads this profile through a **resolution chain**:

```
1. SPARMVET_PROFILE env var       → path to deployment profile YAML
                                    (Galaxy sets this in XML wrapper)
                                    (IRIDA: set at container/process launch)
                                    (Server: set in Docker Compose / systemd unit)
                                    (Local: set in shell profile or .env file)

2. ~/.sparmvet/profile.yaml        → user-level fallback (local PC, no env var)

3. /etc/sparmvet/profile.yaml      → system-level fallback (server, admin-placed)

4. config/connectors/local/        → dev fallback (current behavior)
```

Every deployment context can use the **same Docker image** — only the profile YAML differs, and its path is communicated via `SPARMVET_PROFILE`.

---

### What the deployment profile YAML looks like

For each deployment type:

**Galaxy / general server / local PC (filesystem)**:

```yaml
# galaxy_amr_pipeline.yaml
deployment_type: filesystem
default_manifest: "/data/pipeline/amr/manifest/master.yaml"
default_persona: pipeline-static

locations:
  project_root: "/data/pipeline/amr/"
  raw_data: "inputs/"          # relative to project_root
  manifests: "manifests/"
  curated_data: "parquet/"
  user_sessions: "sessions/"
  gallery: "gallery/"
```

**IRIDA**:

```yaml
# irida_amr_pipeline.yaml
deployment_type: irida
default_manifest: "manifests/amr_master.yaml"   # fetched from IRIDA project
default_persona: pipeline-static

irida:
  base_url: "https://irida.myinstitution.ca"
  project_id: 42
  auth: oauth2                 # token injected via env var SPARMVET_IRIDA_TOKEN
  local_cache: "/tmp/sparmvet_cache/"   # where fetched files land
```

For IRIDA, the **data fetcher module** reads this block and pulls data via REST before the Bootloader hands paths to the rest of the app. The rest of the app (Bootloader, DataOrchestrator, VizFactory) never knows it came from IRIDA — it sees a local cache path like any filesystem deployment.

---

### The "multiple manifests on a general server" case

If a general server hosts multiple pipelines (AMR + Plasmid + Virulence), there are two sub-cases:

**A) One deployment per pipeline** (separate containers/processes, separate `SPARMVET_PROFILE` each) — cleanest, recommended. Each instance is isolated.

**B) One deployment, manifest selector shown** — `default_manifest` is absent from the profile, persona allows manifest switching, user picks from a dropdown of manifests under `locations.manifests`. This is the "admin deploys once, scientists pick their pipeline" model.

Both work with the same resolution chain — the presence or absence of `default_manifest` in the profile controls whether the selector appears (governed by persona gate).

### What needs to change in the current structure

|Component|Change|
|---|---|
|`config/connectors/`|Rename concept to `config/deployment/` or keep as `connectors/` but extend schema|
|Connector template|Add `deployment_type`, `default_manifest`, `default_persona`, `project_root` + relative sub-paths|
|`bootloader.py`|Add `SPARMVET_PROFILE` env var resolution chain (4 levels)|
|New: `libs/connectors/`|Small library with `FilesystemConnector` and `IridaConnector` classes — same interface, different `fetch_data()` implementation|
|`config/connectors/local/local_connector.yaml`|Keep as dev fallback, add `deployment_type: filesystem`|
|Galaxy wrapper (future)|XML sets `SPARMVET_PROFILE=/data/profiles/amr.yaml`|
|IRIDA integration (future)|Same env var, different profile content + IRIDA fetcher|

The `libs/connectors/` library is the **key new piece** — it abstracts the data acquisition layer so the rest of the app remains filesystem-agnostic. This is essentially an adapter pattern: each connector type implements a common interface (`resolve_paths()`, `fetch_data()`, `get_manifest_path()`).
