## Deployment Target Analysis

You have at least **4 deployment contexts**:

|Context|Who launches the app|How config arrives|Data access model|
|---|---|---|---|
|**Galaxy**|Galaxy GxIT wrapper|Env vars injected by XML wrapper|Galaxy job dir, mounted datasets|
|**IRIDA**|IRIDA plugin/iframe|IRIDA's plugin API (likely env vars or URL params)|IRIDA REST API or mounted storage|
|**General server**|sysadmin / Docker Compose / Kubernetes|Env vars or a config file on the filesystem|Network share / NFS / local path configured by admin|
|**Local PC (dev)**|User runs `python app.py`|`local_connector.yaml` as today|Local filesystem|

These are fundamentally different in **how config arrives** and **how data is accessed**, but they all need to land in the same place: the Bootloader knowing which connector + manifest to use.

---

## The Key Insight

These deployment contexts differ in **3 dimensions**:

1. **How the connector path is communicated** (env var / file path / URL param / hardcoded fallback)
2. **How data paths are resolved** (local filesystem / Galaxy mount / IRIDA API / NFS)
3. **Who selects the manifest** (the wrapper/admin / the user / hardcoded)

A single **Configuration Resolution Chain** in the Bootloader can handle all of them if we separate these three concerns cleanly.

### About IRIDA

- Does IRIDA have a mechanism to pass parameters to an external tool at launch (similar to Galaxy env vars)?
- Or does the IRIDA integration look more like: "IRIDA deposits results somewhere, user navigates to the SPARMVET URL separately"?

### About the general server deployment

- Is "general server" a single institution server where an admin deploys once, and multiple users/projects share it? Or one deployment per project?
- Does each project on the server get its own manifest + data directory, or does a single manifest cover all?
- Who sets up the config — the sysadmin once at deploy time, or a user at runtime?

### About local PC

- Is "local PC" always developer mode (you), or could a non-developer scientist run this locally on their own machine?
- Should local PC support multiple manifests (manifest switching UI) or just one at a time?

### Across all contexts

- Should **all four contexts** be able to run the **same Docker image**, just with different config injected? Or are some contexts pip-installed / run directly?
- Is there a concept of a **"project"** that groups manifest + data + user sessions together as a unit? (It sounds like yes — "choose a directory with data access" implies project-scoped data roots)

## Sketch of the Likely Solution

While we finalize the answers, here is the shape of what this is pointing toward:

**A layered configuration resolver in Bootloader**, with a clear priority chain. Each deployment context simply uses a different layer:

```
Priority 1: Environment variable  SPARMVET_CONNECTOR=/path/to/connector.yaml
            → used by Galaxy, IRIDA (if they support env vars), Docker Compose

Priority 2: Config file at a well-known path
            → e.g., /etc/sparmvet/connector.yaml  (sysadmin installs this on server)
            → or    ~/.sparmvet/connector.yaml     (user installs on local PC)

Priority 3: PROJECT_ROOT/config/connectors/local/local_connector.yaml
            → dev fallback (current behavior)
```

The **connector YAML** is the single point of truth for that deployment. It specifies:

- Where data lives (filesystem paths, or API endpoints for IRIDA)
- What the default manifest is (optional — if absent, manifest selector is shown)
- What the default persona is (optional — Galaxy/IRIDA may lock this)
- Where user sessions/exports go

This means:

- **Galaxy**: XML wrapper sets `SPARMVET_CONNECTOR` env var → connector knows Galaxy paths
- **IRIDA**: IRIDA plugin sets `SPARMVET_CONNECTOR` → connector knows IRIDA data paths (or REST API base)
- **Server**: Sysadmin places `/etc/sparmvet/connector.yaml` at deploy time, no env var needed
- **Local PC**: Falls through to `local_connector.yaml` as today, or user places `~/.sparmvet/connector.yaml`

The **manifest question** is separate: each connector optionally declares `default_manifest`. If it does, the manifest is fixed (lower personas never see the selector). If it doesn't, the manifest selector appears (controlled by persona gate).
