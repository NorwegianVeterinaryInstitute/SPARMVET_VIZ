Preface

1.0.1 1. Vision

To build a “Walking Skeleton” for a multi-species (or any other kind of data that can be tabularized) dashboard that is modular, scalable, and easy to maintain.

1.0.1.1 Our first goal

We aimed to employ this dashboard visualize data produced by bioinformatic pipelines hosted:
- on “Use Galaxy” based platform - local “IRIDA” based platform - Or to allow user to upload results produced by galaxy pipelines and visualize them on the dashboard.

As prototyping option we chose: - User upload results produced from a pipeline and associated metadata

The dashboard utilizes Shiny (Express) for Python, Polars for high-performance data handling, and Plotnine for publication-quality, grammar-of-graphics visualizations.

Goals for initial development are indicated by [^1]

1.0.1.2 Feature expectations

We hoped that this work will not only be a deliverable of a short-term project.

Consequently we attempted to build a professional, scalable app. Thus we have built a “Walking Skeleton”, and adopted a Layered Architecture.

The Decoupling approach of the different aspects of the app should improve layer reusability, maintainability while allowing for further extensions. It would also allow to swap to a different UI (user interface) or the Data source (e.g., move from CSVs to a SQL database) without rewriting the core visualization logic.

1.1.2 2.2. Implementation: The “Vizualisation Factory” logic

When the user selects a type of data; eg species: “E. coli” and “AMR” in the Shiny UI, the server performs the following logic:

  1. Lookup: Find E_coli YAML configuration file and respectives categories with columns to keep -> AMR as described in the YAML. (Can be several types of categories)
  2. Retrieve: Get the list columns it will keep["bla_CTX-M", "ndm-1", "tetA"].
  3. Wrangle: Tell Polars to select those specific columns from the main dataset. Eventually tidy transformation eg. long format
  4. Plot: Pass those columns to a Plotnine function designed to handle “Category: AMR.”

1.1.5 Decoupling data preparation and visualisation layers

Shiny is run on a standard server (#REVIEW - need to be decided eg. FastAPI-based) where the “Server” handles the heavy Polars lifting, and only the results (JSON/HTML) are sent to the user’s browser.

We wont use WASM (no havy data manipulation/exploration on browser) because we need scalability.

  • “Server logic” for data preparation (Polars library)
    • getting data
    • filtering unsuable data, tidying data
    • joining with metadata

Advantages: - alows reduce load on visualisation app - better scability in case of implementation of compute heavy features (eg. increase of data volumne). - Polars allows lazy evaluation (allowin)
